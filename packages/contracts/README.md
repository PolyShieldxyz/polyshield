# packages/contracts

Solidity smart contracts for Polyshield — the on-chain core that verifies ZK proofs, custodies
USDC, and tracks commitments, nullifiers, bets, and credits. Built and tested with
[Foundry](https://book.getfoundry.sh/).

Every production contract is a **UUPS implementation behind an `ERC1967Proxy`** (OpenZeppelin
upgradeable). The proxy addresses are the permanent protocol addresses.

> Read [`../../CLAUDE.md`](../../CLAUDE.md), [`../../docs/architecture.md`](../../docs/architecture.md),
> and [`../../docs/zk-design.md`](../../docs/zk-design.md) before changing anything here. The note
> structure, commitment/nullifier formulas, and circuit public inputs are frozen protocol constants.

---

## Layout

```
src/
  Vault.sol                  Core: verifies proofs, USDC custody, betRecords, pendingCredit, feeConfig
  VaultInputs.sol            Domain types (BetRecord, FeeConfig, *PublicInputs) + public-signal assembly
  VaultLogic.sol             Spend-path function bodies (DELEGATECALL library; keeps Vault under EIP-170)
  CommitmentMerkleTree.sol   Poseidon depth-32 append-only tree, rolling 1024-root window (FC-3)
  NullifierRegistry.sol      Spent-nullifier dedup
  PoseidonT3Hasher.sol       BN254 Poseidon hasher used by the tree
  verifiers/                 9 snarkjs-generated Groth16 adapters (one per circuit, slots 0–8)
  interfaces/                ICTF, ICollateralOnramp, ICollateralOfframp, IPoseidonT3, IVerifier
  mocks/                     MockUSDC, MockCTF, MockPUSD, on/offramp, MockDepositWallet, etc. (local dev/tests)
script/                      Foundry deploy + ops scripts (Deploy, DeployLib, UpgradeVault, …)
test/                        Foundry tests (Vault, tree, registry, RealVerifier, Upgrade, gas bench)
```

**EIP-170 note:** the Vault is kept under the 24,576-byte runtime limit by DELEGATECALL-linking two
external libraries (`VaultInputs` + `VaultLogic`). When adding bytecode, run `forge build --sizes`
first; if tight, move the function *body* into `VaultLogic`, not the Vault. See `CLAUDE.md`.

## The 9 verifier slots

| Slot | Verifier | Vault entry point |
|---|---|---|
| 0 | `BetAuthVerifier` | `authorizeBet` |
| 1 | `SettlementCreditVerifier` | `creditSettlement` |
| 2 | `WithdrawalVerifier` | `withdraw` |
| 3 | `BetCancelVerifier` | `betCancellationCredit` |
| 4 | `CancelCreditVerifier` | `naCancellationCredit` |
| 5 | `DepositVerifier` | `deposit` (mandatory binding proof, FC-2) |
| 6 | `PositionCloseVerifier` | `closePosition` (FC-1) |
| 7 | `PartialCreditVerifier` | `partialFillCredit` (FC-4) |
| 8 | `ConsolidateVerifier` | `consolidate` (FC-8) |

Each verifier is itself a UUPS proxy and exposes an owner-only `setBase(address)` to adopt a new
verifying key without a full proxy migration. The Vault swaps a verifier *slot* via the
48h-timelocked `proposeVerifier` / `acceptVerifier`.

## Dependencies (git submodules)

Third-party Solidity libraries are **not vendored into this repo** — they are pinned git submodules
under `lib/` (see [`../../.gitmodules`](../../.gitmodules)). A fresh clone has empty `lib/` dirs until
you fetch them:

```bash
# When cloning the repo:
git clone --recurse-submodules <repo-url>

# Or, if you already cloned without --recurse-submodules:
git submodule update --init --recursive
# (equivalently, from packages/contracts: `forge install`)
```

Pinned versions:

| Submodule | Version | Remapping |
|---|---|---|
| `openzeppelin-contracts` | `v5.6.1` | `@openzeppelin/` |
| `openzeppelin-contracts-upgradeable` | `v5.6.1` | `@openzeppelin-upgradeable/` |
| `forge-std` | `v1.16.1` | `forge-std/` |
| `poseidon-solidity` | pinned commit | `@poseidon-solidity/` |

Each submodule is locked to an exact commit, so every checkout builds against the same library
source. `forge build` / `forge test` will fail with "file not found" import errors if the submodules
have not been fetched.

## Build & test

```bash
cd packages/contracts
forge build                 # or: pnpm contracts:build (from repo root)
forge test                  # or: pnpm contracts:test
forge test --gas-report
forge coverage              # or: pnpm contracts:coverage

# Run the on-chain proof-verification suite against real Groth16 proofs:
forge test --match-contract RealVerifierTest
```

`RealVerifier.t.sol` consumes the `test/fixtures/*_proof.json` files generated by the circuit
pipeline (`pnpm circuits:all` from the repo root). After regenerating verifiers or fixtures, rerun
this suite — a fresh trusted setup changes every verifying key.

**Toolchain (`foundry.toml`):** `solc 0.8.24`, `evm_version = "cancun"` (EIP-1153 transient storage
for `ReentrancyGuardTransient` — supported on Polygon since the Napoli hardfork), `via_ir = false`,
1000 fuzz runs.

## Deployment

Deploy through the proxy pattern in `script/`:

- `Deploy.s.sol` / `MockDeploy.s.sol` — full deployment (uses `DeployLib.sol`, which predicts the
  Vault proxy address to resolve the Vault ↔ Tree ↔ Registry init cycle).
- `DeployPoseidon.s.sol`, `DeployVerifiers.s.sol`, `AcceptVerifiers.s.sol` — staged pieces.
- `UpgradeVault.s.sol` — UUPS upgrade of the Vault implementation.
- `SetDeploymentCap.s.sol`, `Set*VerifierBases.s.sol` — ops levers.

Constructors are replaced by `initialize(...)` (`initializer`-guarded); implementations carry
`constructor() { _disableInitializers(); }`. Storage is append-only behind a trailing `__gap` —
**never reorder or insert existing state across an upgrade.** `CommitmentMerkleTree`'s storage
layout is frozen.

## Security properties enforced here

- ZK proof verification gates **every** state transition (no admin bypass of the proof check).
- Nullifier registry check happens before any state change (checks-effects-interactions).
- Merkle root window: accepts the last 1024 roots (O(1) membership, FC-3).
- W-to-W withdrawal: recipient is bound to the note's `owner_address` both in-circuit and in `withdraw`.
- `ReentrancyGuardTransient` on state-modifying functions.
- `_authorizeUpgrade` is `onlyOwner`, **instant (no timelock)** — the largest trust assumption; the
  owner role MUST be a multisig/HSM in production. See `docs/threat-model.md` (T-UPGRADE / T21).

See [`../../docs/threat-model.md`](../../docs/threat-model.md) for the full attack surface.
