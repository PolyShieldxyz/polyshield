# Q16 — Proving Backend Comparison: UltraPLONK vs. Groth16

**Purpose:** Decision support document for mainnet proving backend selection.  
**Audience:** Polyshield team + managers.  
**Status:** Decision pending. See open-questions.md Q16.  
**Date:** 2026-05-23  

---

## 1. What This Decision Is

Every ZK proof Polyshield generates (bet authorization, settlement credit, withdrawal, cancellation) must be verified on-chain by a Solidity verifier contract. The verifier contract is generated from the chosen proving backend. Once deployed and in use, changing the backend requires deploying new verifier contracts and migrating all existing commitments — a protocol-level breaking change. This decision locks in before mainnet deployment.

The two candidates are:

- **UltraPLONK** — the native Noir/Barretenberg backend. Currently used in dev and all benchmarking.
- **Groth16** — the industry-standard pairing-based backend. Requires a per-circuit trusted setup ceremony. Used in Tornado Cash, Zcash Sapling, and most production ZK apps to date.

UltraHonk was previously evaluated and has been permanently dropped — it is not a candidate.

## 4. What Each Backend Requires from the Team

### Groth16

**One-time ceremony (before mainnet):**

- Run a Powers of Tau ceremony for each of the 5 circuits, or reuse an existing trusted ceremony (e.g., Hermez/Polygon's ptau28).
- This produces 5 `.zkey` files (~8.5–8.7 MB each, ~43 MB total). These are the proving keys.
- The ceremony can be done offline by the team. It does not affect users.
- The `.zkey` files are hosted on a CDN and fetched by the browser on demand (lazy-loaded per action).

**Ongoing toolchain:**

- `snarkjs` (673 KB JS library, served to browser) replaces `@aztec/bb.js` (36 MB WASM) as the browser proving engine.
- Per-circuit witness-generator WASM files (~2.3 MB each) are fetched lazily per action.
- Verifier contracts are generated by `snarkjs` and are already present in `packages/groth16/` as benchmarking artifacts.

**Trust assumption:**

- Anyone who participated in or compromised the ceremony could theoretically forge proofs. In practice, reusing a large public ceremony (Hermez ptau28 had thousands of contributors) makes this negligible.

### UltraPLONK

**No ceremony required.** The Barretenberg SRS (Aztec's Ignition ceremony, run once universally) is embedded in the `@aztec/bb.js` WASM. No team action needed.

**Ongoing toolchain:**

- `@aztec/bb.js` WASM (36 MB, loaded once per browser session, cached thereafter).
- Circuit artifact JSONs (1.7 MB each) fetched lazily per action.
- Verifier contracts auto-generated by `nargo` — already in use in the current dev stack.

## 5. Browser Memory Footprint

This is the key operational concern for user experience.


| Asset                                           | Groth16                      | UltraPLONK                      |
| ----------------------------------------------- | ---------------------------- | ------------------------------- |
| Proving engine (downloaded once)                | snarkjs.min.js = **673 KB**  | barretenberg.wasm = **36 MB**   |
| Per-circuit proving key (downloaded per action) | .zkey = **~8.7 MB**          | circuit .json = **~1.7 MB**     |
| Per-circuit witness WASM                        | .wasm = **~2.3 MB**          | (included in bb.js)             |
| **Total download per action (first use)**       | **~11.7 MB**                 | **~37.7 MB**                    |
| **Total download per action (cached session)**  | **~11 MB** (zkey per action) | **~1.7 MB** (circuit json only) |
| **Peak RAM during proving**                     | ~40–60 MB                    | ~100–150 MB                     |


Key insight: after the first page load, UltraPLONK is actually lighter per-action (only the 1.7 MB circuit JSON needs loading, since bb.js is cached). Groth16 must fetch a fresh ~8.7 MB `.zkey` for each new circuit type the user encounters in a session.

On mobile, the 100–150 MB peak RAM for UltraPLONK is the more serious concern. Groth16's ~40–60 MB peak is materially safer on low-end devices.

## 6. Proving Time UX Impact


| Backend    | Typical prove time | User perception                          |
| ---------- | ------------------ | ---------------------------------------- |
| Groth16    | ~1.0–1.1 seconds   | Fast — a spinner the user barely notices |
| UltraPLONK | ~2.3–2.6 seconds   | Noticeable — user sees a 2–3 second wait |


For a trading app where users may place multiple bets in a session, the cumulative difference matters. 5 bets = ~5 seconds (Groth16) vs. ~12 seconds (UltraPLONK). Both are far below the 30-second threshold where users perceive the app as broken, but Groth16 is a meaningfully better UX.

## 7. Architectural Trade-offs


| Dimension               | Groth16                              | UltraPLONK                                        |
| ----------------------- | ------------------------------------ | ------------------------------------------------- |
| Trusted setup           | Per-circuit ceremony (one-time)      | Universal — no ceremony                           |
| Trust assumption        | Ceremony participants                | None beyond Aztec's Ignition ceremony (universal) |
| Proof size              | 256 B (constant)                     | 2,144 B (constant)                                |
| Verifier gas            | ~35–70k gas                          | ~437–440k gas                                     |
| Proving time            | ~1.0–1.1 s                           | ~2.3–2.6 s                                        |
| Peak browser RAM        | ~40–60 MB                            | ~100–150 MB                                       |
| Per-action CDN download | ~11 MB (.zkey)                       | ~1.7 MB (.json, after engine cached)              |
| First-session download  | ~11.7 MB                             | ~37.7 MB                                          |
| Noir support            | Via `bb` + snarkjs pipeline          | Native `nargo`                                    |
| Recursion / aggregation | Requires wrapper circuits            | Native support                                    |
| Circuit upgrade process | New ceremony per circuit             | Recompile and redeploy verifier                   |
| Production precedent    | Tornado Cash, Zcash, most ZK bridges | Newer; Aztec ecosystem primarily                  |


---

## 8. Recommendation

Groth16 is the stronger choice for a trading app where UX latency and mobile memory are first-class concerns. The ceremony overhead is a one-time operational cost — not a recurring burden — and can be mitigated by reusing the Hermez ptau28 ceremony, which is already present in `Benchmarking/groth16/setup/`. Groth16's on-chain gas advantage (6x cheaper verification) directly reduces the relay gas fee passed to users on every bet.

UltraPLONK is the right choice if ceremony operations are unacceptable, or if future circuit aggregation (batching multiple users' proofs into one on-chain call) is a near-term priority — UltraPLONK's native recursion support makes that significantly easier to implement.

**The decision is pending Arya's consultation with management.** No implementation work should begin on mainnet verifier contracts until this is resolved. Dev stack remains UltraPLONK regardless of outcome.
