# ZK Circuit Design: Polyshield

**Version:** 0.1 (Design Phase)
**Status:** Specifications written. No Noir code yet.
**ZK Language:** Noir (Aztec)
**ZK Backend:** UltraPLONK (dev/testing); mainnet backend TBD — UltraPLONK vs. Groth16 under evaluation (see open-questions.md Q16)

---

## 1. Primitives

### Hash Function: Poseidon

All commitments and nullifiers use Poseidon. Rationale: native Noir support, ZK-efficient (few constraints vs. SHA-256), standard in modern ZK note systems (Tornado Cash Nova, Zcash Orchard).

Poseidon variant: `poseidon2` (2 inputs) and `poseidon3` (3 inputs) as provided by `noir-stdlib`.

### Note Structure

A user's balance state is represented as a "note":

```
Note {
  secret: Field,      // 254-bit random value, generated by user, never leaves browser
  balance: u64,       // current balance in USDC micro-units (6 decimals)
  nonce: u64,         // increments with every state transition (bet, credit, partial withdraw)
}
```

### Commitment

```
commitment = poseidon3(secret, balance, nonce)
```

Stored on-chain in the Commitment Merkle Tree. The commitment does NOT reveal secret, balance, or nonce.

### Nullifier

```
nullifier = poseidon2(secret, nonce)
```

Nullifiers are published on-chain when a note is spent (when a new note replaces it). The nullifier does not reveal which commitment it belongs to (it does not contain the balance, only the secret and nonce). This prevents double-spending while maintaining privacy.

### Merkle Tree

- Hash function: Poseidon (binary tree, `poseidon2(left, right)`)
- Depth: 32 (supports 2^32 leaves)
- Type: Append-only. Old leaves are never deleted. Spent leaves are invalidated via nullifiers.
- Implementation: Incremental Merkle Tree in Solidity (similar to Tornado Cash's `MerkleTreeWithHistory`).

## 2. Circuit 1: Bet Authorization

**File:** `packages/circuits/bet_auth.nr`
**Purpose:** Proves the user has sufficient balance and authorizes the vault to execute a specific bet on their behalf, without revealing which depositor they are.

### Inputs

```noir
// Private inputs (known only to the user)
secret: Field,
current_balance: u64,
nonce: u64,
merkle_path: [Field; 32],    // sibling hashes from leaf to root
merkle_path_indices: [u1; 32], // 0 = left child, 1 = right child

// Public inputs (revealed on-chain)
merkle_root: Field,          // current root of the Commitment Merkle Tree
nullifier: Field,            // = poseidon2(secret, nonce)
new_commitment: Field,       // = poseidon3(secret, current_balance - bet_amount, nonce + 1)
bet_amount: u64,             // pUSD amount being bet (micro-units, 6 decimals)
price: u64,                  // limit price as a fixed-point integer (e.g. 0.65 = 65_000_000 in 1e8 units)
expected_shares: u64,        // = floor(bet_amount / price) — verified in circuit
market_id: Field,            // Polymarket condition ID
outcome_side: u8,            // 0 = NO, 1 = YES (or index for multi-outcome markets)
position_id: Field,          // ERC-1155 positionId for the outcome token
```

**Note on `price` and `expected_shares`:** Added as part of the Q4 resolution. `price` is a fixed-point number at 1e8 precision (e.g. a price of 0.65 is represented as 65_000_000). `expected_shares` is constrained in the circuit to equal `floor(bet_amount * 1e8 / price)`. The Vault records `(nullifier_hash, market_id, position_id, expected_shares)` on-chain at `authorizeBet` time. The Settlement Credit circuit reads `expected_shares` from Vault storage rather than accepting a self-reported value.

**Note on `fee` (FC-10 — IMPLEMENTED, live circuit only).** The live Circom `groth16/bet_auth.circom` adds a **10th public input `fee`**, Vault-injected from `feeConfig`: `fee = bet_amount * betFeeBps / 10000 + relayGasFeeUSDC`. The balance constraint becomes `new_balance = current_balance - bet_amount - fee`. Because the Vault — not the user — supplies `fee` as a public input (like the injected `bet_amount` for cancellations), a proof built with any other fee fails verification. The full public-signal order is `[merkle_root, nullifier, new_commitment, bet_amount, price, expected_shares, market_id, outcome_side, position_id, fee]`. (This `.nr` file is a spec reference and still shows the pre-fee 9-input, 3-field-note form; the authoritative source is `packages/circuits/groth16/bet_auth.circom`.)

### Constraints (Pseudocode)

```noir
fn main(
  secret: Field,
  current_balance: u64,
  nonce: u64,
  merkle_path: [Field; 32],
  merkle_path_indices: [u1; 32],
  merkle_root: pub Field,
  nullifier: pub Field,
  new_commitment: pub Field,
  bet_amount: pub u64,
  price: pub u64,
  expected_shares: pub u64,
  market_id: pub Field,
  outcome_side: pub u8,
  position_id: pub Field,
) {
  // 1. Verify the old note commitment is in the Merkle tree
  let old_commitment = poseidon3(secret, current_balance as Field, nonce as Field);
  let computed_root = merkle_verify(old_commitment, merkle_path, merkle_path_indices);
  assert(computed_root == merkle_root);

  // 2. Verify the nullifier is correct
  let computed_nullifier = poseidon2(secret, nonce as Field);
  assert(computed_nullifier == nullifier);

  // 3. Verify the user has sufficient balance
  assert(current_balance >= bet_amount);

  // 4. Verify expected_shares = floor(bet_amount * PRICE_PRECISION / price)
  //    PRICE_PRECISION = 100_000_000 (1e8)
  //    This prevents the user from claiming more shares than the price justifies.
  let computed_shares = (bet_amount as u128 * 100_000_000u128) / (price as u128);
  assert(expected_shares as u128 == computed_shares);

  // 5. Verify the new commitment is correct (balance decremented, nonce incremented)
  let new_balance = current_balance - bet_amount;
  let computed_new_commitment = poseidon3(secret, new_balance as Field, (nonce + 1) as Field);
  assert(computed_new_commitment == new_commitment);

  // market_id, outcome_side, and position_id are public inputs bound to this proof.
  // The Signing Layer will only execute a bet matching exactly these parameters as
  // recorded in the on-chain BetAuthorized event.
}
```

### Notes on Design

Adding `price`, `expected_shares`, and `position_id` as public inputs resolves Q4: the Vault now records `expected_shares` on-chain at authorizeBet time, and the Settlement Credit proof uses that value rather than self-reported share counts.

The `market_id`, `outcome_side`, `bet_amount`, `price`, and `expected_shares` being public means an on-chain observer can see "someone in the vault bet on market X, side Y, Z pUSD at price P, expecting S shares." The depositor's identity is hidden but bet activity is visible. This is the **bet descriptor privacy** open question (Q6). Encrypting these values requires the Signing Layer to have a decryption key, which complicates v1. The current design accepts visibility in v1.

## 3. Circuit 2: Settlement Credit

**File:** `packages/circuits/settlement_credit.nr`
**Purpose:** Proves the user owns a winning bet position and is entitled to credit their note balance with the payout.

### Inputs

```noir
// Private inputs (known only to the user)
secret: Field,
balance_before_credit: u64,
nonce: u64,
merkle_path: [Field; 32],
merkle_path_indices: [u1; 32],

// Public inputs (revealed on-chain)
merkle_root: Field,
nullifier: Field,             // = poseidon2(secret, nonce)
new_commitment: Field,        // = poseidon3(secret, balance_before_credit + total_credit, nonce + 1)
nullifier_of_bet: Field,      // nullifier of the Bet Authorization note being settled (lookup key)
market_id: Field,             // must match a resolved market in the Indexer
payout_per_share: u64,        // from Polymarket CTF contract: payoutNumerators[i] / payoutDenominator
shares_held: u64,             // NOT user-supplied — read by Vault from betRecords[nullifier_of_bet].expected_shares
total_credit: u64,            // = shares_held * payout_per_share
```

**Key change from Q4 resolution:** `shares_held` is now a **public** input, not a private one. It is NOT supplied by the user in proof calldata. Instead, the Vault's `creditSettlement()` function reads it from its own storage (`betRecords[nullifier_of_bet].expected_shares`, populated at `authorizeBet` time) and passes it directly to the ZK verifier. The user cannot inflate this value.

`nullifier_of_bet` is how the user identifies which bet they are claiming settlement for. The Vault uses it as the key into `betRecords`. The user knows this value because they computed it at bet authorization time from their `secret` and the bet's `nonce`.

### Constraints (Pseudocode)

```noir
fn main(
  secret: Field,
  balance_before_credit: u64,
  nonce: u64,
  merkle_path: [Field; 32],
  merkle_path_indices: [u1; 32],
  merkle_root: pub Field,
  nullifier: pub Field,
  new_commitment: pub Field,
  nullifier_of_bet: pub Field,
  market_id: pub Field,
  payout_per_share: pub u64,
  shares_held: pub u64,        // supplied by Vault from betRecords[nullifier_of_bet], not by user
  total_credit: pub u64,
) {
  // 1. Verify the settlement note is in the Merkle tree
  let old_commitment = poseidon3(secret, balance_before_credit as Field, nonce as Field);
  let computed_root = merkle_verify(old_commitment, merkle_path, merkle_path_indices);
  assert(computed_root == merkle_root);

  // 2. Verify the nullifier of the current (settlement) note
  let computed_nullifier = poseidon2(secret, nonce as Field);
  assert(computed_nullifier == nullifier);

  // 3. Verify total_credit = shares_held * payout_per_share
  // shares_held was set by the Vault at authorizeBet time — the circuit
  // simply enforces the arithmetic; it cannot be manipulated by the user.
  assert(total_credit == (shares_held as u128 * payout_per_share as u128) as u64);

  // 4. Verify the new commitment (balance credited, nonce incremented)
  let new_balance = balance_before_credit + total_credit;
  let computed_new_commitment = poseidon3(secret, new_balance as Field, (nonce + 1) as Field);
  assert(computed_new_commitment == new_commitment);

  // nullifier_of_bet and market_id are public inputs used by the Vault to look up
  // the bet record. The circuit does not constrain nullifier_of_bet against secret —
  // the Vault enforces consistency by checking betRecords[nullifier_of_bet].market_id
  // matches the supplied market_id before calling the verifier.
}
```

### Notes on Design

The `creditSettlement()` call sequence in `Vault.sol`:
1. Look up `betRecords[inputs.nullifier_of_bet]`. Revert if not found.
2. Assert `betRecords[inputs.nullifier_of_bet].market_id == inputs.market_id`. Revert if mismatch.
3. Assert the market is resolved (check Indexer registry or on-chain CTF state).
4. Inject `betRecords[inputs.nullifier_of_bet].expected_shares` as the `shares_held` public input before calling the verifier. The user cannot supply a different value here.
5. Verify the ZK proof with the assembled public inputs.
6. Mark nullifier spent, insert new commitment, emit `SettlementCredited`.

`payout_per_share` is derived from CTF on-chain data (`payoutNumerators[i] / payoutDenominator`) and can be verified by the Vault contract directly against the CTF contract before accepting the proof, removing any oracle trust.

## 4. Circuit 3: Withdrawal

**File:** `packages/circuits/withdrawal.nr`
**Purpose:** Proves the user holds a valid note with sufficient balance to withdraw, and binds the withdrawal to a specific recipient address (preventing front-running).

### Inputs

```noir
// Private inputs
secret: Field,
final_balance: u64,
nonce: u64,
merkle_path: [Field; 32],
merkle_path_indices: [u1; 32],
recipient_address: Field,    // actual withdrawal address, kept private

// Public inputs
merkle_root: Field,
nullifier: Field,
withdrawal_amount: u64,
recipient_hash: Field,       // = poseidon2(recipient_address)
```

### Constraints (Pseudocode)

```noir
fn main(
  secret: Field,
  final_balance: u64,
  nonce: u64,
  merkle_path: [Field; 32],
  merkle_path_indices: [u1; 32],
  recipient_address: Field,
  merkle_root: pub Field,
  nullifier: pub Field,
  withdrawal_amount: pub u64,
  recipient_hash: pub Field,
) {
  // 1. Verify the note is in the Merkle tree
  let commitment = poseidon3(secret, final_balance as Field, nonce as Field);
  let computed_root = merkle_verify(commitment, merkle_path, merkle_path_indices);
  assert(computed_root == merkle_root);

  // 2. Verify the nullifier
  let computed_nullifier = poseidon2(secret, nonce as Field);
  assert(computed_nullifier == nullifier);

  // 3. Verify withdrawal amount does not exceed balance
  assert(withdrawal_amount <= final_balance);

  // 4. Verify the recipient hash (prevents front-running by binding recipient inside proof)
  let computed_hash = poseidon2(recipient_address, 0); // domain separation TBD
  assert(computed_hash == recipient_hash);
}
```

### Notes on Design

The `recipient_address` is private. The contract receives `recipient_hash` and the actual `recipientAddress` as separate arguments, then verifies `poseidon2(recipientAddress) == recipient_hash` before transferring. This prevents MEV bots from replacing the recipient address in the mempool.

If `withdrawal_amount < final_balance`, the remaining balance is forfeit in the current design. A "partial withdrawal" flow that creates a new note with the remaining balance is a future enhancement (requires a new proof type: Withdrawal with Change).

## 5. Circuit 4: Bet Cancellation Credit

**File:** `packages/circuits/bet_cancel.nr`
**Purpose:** Restores `bet_amount` to a user's note when a FOK order fails to fill after `authorizeBet` has already been verified on-chain. The user's note was debited but no Polymarket bet was placed.

**Trigger mechanism:** The Signing Layer operator calls `Vault.reportFOKFailure(nullifier_of_bet)`, which sets `betRecords[nullifier_of_bet].status = FAILED` and emits a `FOKFailed` event. The user (or their relay) observes the event and submits this proof to restore their balance.

**Why `reportFOKFailure` must be operator-restricted:** If anyone could call `reportFOKFailure`, they could falsely mark a filled bet as failed and allow the user to credit back funds that were legitimately spent. Only the registered `signingLayerOperator` address may call it.

### Inputs

```noir
// Private inputs
secret: Field,
current_balance: u64,        // balance in the post-bet note (= original_balance - bet_amount)
nonce: u64,                  // nonce of the post-bet note (= original_nonce + 1)
merkle_path: [Field; 32],
merkle_path_indices: [u1; 32],

// Public inputs
merkle_root: Field,
nullifier: Field,            // = poseidon2(secret, nonce)  -- nullifier of the post-bet note
new_commitment: Field,       // = poseidon3(secret, current_balance + bet_amount, nonce + 1)
nullifier_of_bet: Field,     // lookup key into betRecords — identifies the failed bet
bet_amount: u64,             // NOT user-supplied — read from betRecords[nullifier_of_bet].bet_amount by the Vault
```

### Constraints (Pseudocode)

```noir
fn main(
  secret: Field,
  current_balance: u64,
  nonce: u64,
  merkle_path: [Field; 32],
  merkle_path_indices: [u1; 32],
  merkle_root: pub Field,
  nullifier: pub Field,
  new_commitment: pub Field,
  nullifier_of_bet: pub Field,
  bet_amount: pub u64,     // injected by Vault from betRecords, not user-supplied
) {
  // 1. Verify the post-bet note is in the Merkle tree
  let old_commitment = poseidon3(secret, current_balance as Field, nonce as Field);
  let computed_root = merkle_verify(old_commitment, merkle_path, merkle_path_indices);
  assert(computed_root == merkle_root);

  // 2. Verify the nullifier of the post-bet note
  let computed_nullifier = poseidon2(secret, nonce as Field);
  assert(computed_nullifier == nullifier);

  // 3. Verify the new commitment restores exactly bet_amount to the balance
  let restored_balance = current_balance + bet_amount;
  let computed_new_commitment = poseidon3(secret, restored_balance as Field, (nonce + 1) as Field);
  assert(computed_new_commitment == new_commitment);

  // nullifier_of_bet is used by the Vault to look up the failed bet record.
  // The circuit does not constrain nullifier_of_bet against secret or nonce —
  // the Vault enforces consistency by checking betRecords[nullifier_of_bet].status == FAILED
  // and injecting bet_amount from storage before calling the verifier.
}
```

### Vault Call Sequence for `betCancellationCredit()`

1. Look up `betRecords[inputs.nullifier_of_bet]`. Revert if not found.
2. Assert `betRecords[inputs.nullifier_of_bet].status == FAILED`. Revert otherwise (prevents use before FOK failure is reported, and prevents double-credit after `CANCELLED_CREDITED`).
3. Inject `betRecords[inputs.nullifier_of_bet].bet_amount` as the `bet_amount` public input. The user cannot supply a different value.
4. Verify the ZK proof with the assembled public inputs.
5. Mark `inputs.nullifier` as spent in the nullifier registry. Insert `inputs.new_commitment` into the Merkle tree.
6. Set `betRecords[inputs.nullifier_of_bet].status = CANCELLED_CREDITED`. Emit `BetCancellationCredited`.

### Notes on Design

The constraint logic of this circuit is structurally identical to N/A Cancellation Credit (Section 6). The only difference is the Vault-side trigger check: here it requires `status == FAILED`, set by the Signing Layer operator. In the N/A circuit, the Vault checks the CTF contract for N/A resolution.

`nullifier_of_bet` is the nullifier of the original pre-bet note (computed as `poseidon2(secret, original_nonce)` at bet authorization time). It is distinct from `nullifier` in this circuit, which is the nullifier of the post-bet note (`poseidon2(secret, original_nonce + 1)`).

## 6. Circuit 5: N/A Cancellation Credit

**File:** `packages/circuits/cancel_credit.nr`
**Purpose:** Restores `bet_amount` to a user's note when Polymarket resolves a market as N/A (cancelled), returning all bets in full. The CTF contract signals N/A resolution by setting all `payoutNumerators` to zero for the condition.

**No operator action required.** Unlike Bet Cancellation Credit, the trigger here is purely on-chain CTF state. The Vault reads `CTF.payoutNumerators(conditionId)` directly to verify N/A status. No trusted operator call is needed.

### Inputs

```noir
// Private inputs
secret: Field,
current_balance: u64,        // balance after the bet was authorized (= original_balance - bet_amount)
nonce: u64,                  // nonce of the post-bet note
merkle_path: [Field; 32],
merkle_path_indices: [u1; 32],

// Public inputs
merkle_root: Field,
nullifier: Field,            // = poseidon2(secret, nonce)
new_commitment: Field,       // = poseidon3(secret, current_balance + bet_amount, nonce + 1)
nullifier_of_bet: Field,     // lookup key into betRecords
market_id: Field,            // must match betRecords[nullifier_of_bet].market_id (Vault checks this)
bet_amount: u64,             // NOT user-supplied — read from betRecords[nullifier_of_bet].bet_amount by the Vault
```

### Constraints (Pseudocode)

```noir
fn main(
  secret: Field,
  current_balance: u64,
  nonce: u64,
  merkle_path: [Field; 32],
  merkle_path_indices: [u1; 32],
  merkle_root: pub Field,
  nullifier: pub Field,
  new_commitment: pub Field,
  nullifier_of_bet: pub Field,
  market_id: pub Field,
  bet_amount: pub u64,     // injected by Vault from betRecords, not user-supplied
) {
  // 1. Verify the post-bet note is in the Merkle tree
  let old_commitment = poseidon3(secret, current_balance as Field, nonce as Field);
  let computed_root = merkle_verify(old_commitment, merkle_path, merkle_path_indices);
  assert(computed_root == merkle_root);

  // 2. Verify the nullifier
  let computed_nullifier = poseidon2(secret, nonce as Field);
  assert(computed_nullifier == nullifier);

  // 3. Verify the new commitment restores exactly bet_amount to the balance
  let restored_balance = current_balance + bet_amount;
  let computed_new_commitment = poseidon3(secret, restored_balance as Field, (nonce + 1) as Field);
  assert(computed_new_commitment == new_commitment);
}
```

### Vault Call Sequence for `naCancellationCredit()`

1. Look up `betRecords[inputs.nullifier_of_bet]`. Revert if not found.
2. Assert `betRecords[inputs.nullifier_of_bet].market_id == inputs.market_id`.
3. Resolve the `conditionId` for `market_id` from the Indexer registry (or derive it on-chain). Call `CTF.payoutNumerators(conditionId)` and verify all values are zero — the on-chain fingerprint of an N/A resolution. Revert if not N/A.
4. Inject `betRecords[inputs.nullifier_of_bet].bet_amount` as the `bet_amount` public input.
5. Verify the ZK proof.
6. Mark nullifier spent, insert new commitment. Set `betRecords[inputs.nullifier_of_bet].status = CANCELLED_CREDITED`. Emit `NACancellationCredited`.

### Notes on Design

The constraint structure is identical to Bet Cancellation Credit. They are kept as separate circuits (separate compiled artifacts, separate verifier contracts) per the Q8 resolution: distinct trigger semantics deserve distinct proof types, making the audit surface cleaner.

A single user can only claim one cancellation credit per bet record. The `CANCELLED_CREDITED` status prevents double-claiming regardless of which circuit path was used first.

## 7. Merkle Path Verification (Shared Library)

Used in all five circuits.

Used in all three circuits.

```noir
fn merkle_verify(
  leaf: Field,
  path: [Field; 32],
  indices: [u1; 32],
) -> Field {
  let mut current = leaf;
  for i in 0..32 {
    if indices[i] == 0 {
      current = poseidon2(current, path[i]); // current is left child
    } else {
      current = poseidon2(path[i], current); // current is right child
    }
  }
  current // computed root
}
```

This is a standard incremental Merkle tree verification. The circuit proves the leaf is at the correct position in the tree with the given siblings.

## 8. Gas Cost Estimates (Preliminary)

| Proof Type | Proof size (Groth16 ~200B / UltraPLONK ~2KB) | Estimated on-chain verify gas |
|---|---|---|
| Bet Authorization | ~200 bytes | ~250,000 gas |
| Settlement Credit | ~200 bytes | ~250,000 gas |
| Withdrawal | ~200 bytes | ~250,000 gas |
| Bet Cancellation Credit | ~200 bytes | ~250,000 gas |
| N/A Cancellation Credit | ~200 bytes | ~250,000 gas |

At Polygon gas prices (~30-100 gwei) and MATIC price (~$0.50-$1.00), verification cost per operation is approximately $0.004-$0.025. Acceptable for bets above ~$50 USDC.

For very small bets, a proof batching/recursion layer (Nova or ProtoGalaxy folding) could amortize verification costs across multiple users' proofs in a single on-chain call. This is a v2+ enhancement.

## §X — Consolidate (FC-8): multi-note merge

`packages/circuits/groth16/consolidate.circom` (`Consolidate(K)`, K=4). Merges up to 4 **same-owner** notes into one, continuing slot 0's lineage. Pure value-preserving merge — no bet, no withdrawal, no token movement. Used by the frontend to defragment notes before a bet/withdrawal that exceeds any single note's balance.

**Private inputs (per slot j=0..3):** `secret[j], balance[j], nonce[j], merkle_path[j][32], merkle_path_indices[j][32], is_active[j]`; plus a single shared `owner_address` (all of a wallet's notes carry the same owner_address).

**Public inputs (6):** `merkle_root, nullifier[0..3], new_commitment`. Verifier slot `CONSOLIDATE = 8`.

**Constraints (per slot j):**
- `AssertBool(is_active[j])` — strict boolean.
- `AssertBits(64)` on `balance[j]`, `nonce[j]`.
- `cm[j] = Poseidon4(secret[j], balance[j], nonce[j], owner_address)`.
- **Gated membership:** `root_j = PoseidonMerklePath(32)(cm[j], …)`; `is_active[j] * (root_j − merkle_root) === 0` (active ⇒ in tree; inactive ⇒ unconstrained).
- **Gated nullifier:** `nullifier[j] === is_active[j] * Poseidon2(secret[j], nonce[j])` (active ⇒ real nullifier; inactive ⇒ 0, the Vault's skip sentinel).
- **Gated value:** `eff[j] = is_active[j] * balance[j]`.

**Aggregate:** `is_active[0] === 1`; `sum = Σ eff[j]` (range-checked u64); output `new_commitment === Poseidon4(secret[0], sum, nonce[0]+1, owner_address)`.

**Padding soundness.** (a) inject-value: blocked — inactive `eff[j]=0` (boolean-gated). (b) membership-bypass: blocked — active slot forces `root_j == merkle_root`, and the Vault checks `isKnownRoot`. (c) double-count one note in two active slots: blocked **on-chain** — both slots publish the same nullifier and the second `markSpent` reverts `AlreadySpent`, so the Vault marks every non-zero nullifier without de-duplication. (d) all-inactive forge: blocked by `is_active[0] === 1`. ~34k constraints (well under the 2^17 ptau).

## §Y — Operator fill attestations (FC-9)

Operator fill reporting is now **off-chain and gasless** (no `report*` on-chain functions). The operator signs an EIP-712 `OperatorAttestation{nullifierOfBet, reportType(1=FILLED,2=FAILED,3=PARTIAL,4=SOLD), amountA, amountB}` and the user submits it with their credit proof; the Vault recovers the signer (`ECDSA.recover == signingLayerOperator`) and injects the attested values. The credit circuits are UNCHANGED — they still take the injected amount as a public input; only the SOURCE of that injected value moved from on-chain `betRecords`/`report*` to the operator's signed attestation. See `docs/future-changes.md` FC-9 and `docs/threat-model.md` T22/T23 (single-terminal-signing invariant).
